01
Overview & Philosophy
Why Docker changes everything about how you build software

Before Docker, deploying a Django application meant wrestling with Python versions, OS-level library differences, and the classic nightmare: "it works on my machine, but not in production." Docker solves this by packaging your entire application — Python runtime, dependencies, OS libraries — into a single portable unit called a container.

Core Concept

What is a Container?

A container is a lightweight, isolated process that runs on top of your host operating system. It is not a virtual machine — there is no separate OS kernel. Instead, it shares your host OS kernel but runs in an isolated filesystem, process space, and network. This makes it extremely fast to start (milliseconds) and very small (megabytes, not gigabytes).

Think of it as a shipping container: before shipping containers, every cargo ship was loaded differently, every port had different equipment. After standardisation, one container fits every ship, every crane, every truck. Docker does the same for software environments.

📚 Mental Model

A Dockerfile is the recipe. A Docker image is the baked cake. A Docker container is a slice of that cake being served. You bake once, serve anywhere — locally, in CI, in production — and every slice is identical.

✕ Without Docker

Environment Chaos

Developer A has Python 3.9, Developer B has 3.11. Production runs 3.8. A library works on macOS but fails on Linux. Every new developer spends a day setting up their environment. Bugs exist only in production.

✓ With Docker

Reproducible Environments

Every developer, CI server, and production host runs the exact same image. One command to start. Onboarding takes minutes. "Works on my machine" becomes "works in the container" — and the container runs everywhere.

📚
How to use this manual

Read each chapter in order the first time to understand the why behind every decision. On future projects, use it as a reference — jump to any chapter and copy the verified code blocks. Every mistake you can make has been documented in Chapter 11.

02
Prerequisites
Every tool you need and why you need it

You do not need Python installed locally. You do not need Django installed locally. That is the point. The only things you need on your host machine are the tools that manage Docker, not the tools that run your application.

ToolWhy you need itStatus
Docker Desktop The Docker engine itself. Builds images and runs containers on your local machine. Also includes Docker Compose V2 built in. Required
Git Version control. Required to push code to GitHub and trigger the CI pipeline. Also used by Docker during some multi-stage builds. Required
GitHub Account Hosts your code repository and runs your automated tests via GitHub Actions every time you push a commit. Required
Docker Hub Account The public registry where Docker pulls base images from. Anonymous pulls are rate-limited; logging in from CI prevents build failures. Required
VS Code The recommended editor. Install the Docker extension for container management and syntax highlighting inside Dockerfiles. Recommended
Why no local Python?

The Container is your Environment

A common beginner mistake is installing Python and Django locally and also using Docker. This creates confusion: which Python runs your app? Which pip installs packages?

The rule is simple: your application never runs outside the container. Every Django command — migrate, test, startapp — is run inside a container. Your host machine is only used to invoke Docker commands. This is covered fully in Chapter 8.

⚠️
Docker Hub Rate Limits

Anonymous image pulls are capped at 100 per 6 hours per IP. In CI environments, multiple pipelines running in parallel from shared GitHub-hosted runners will share the same IP and hit this limit. Always authenticate with Docker Hub in your CI pipeline (Chapter 9 shows exactly how to do this).

03
Project Structure
The layout every Docker-Django project must follow

Consistency in project layout is not a matter of preference — it is a requirement when Docker is involved. The Dockerfile contains explicit COPY instructions that reference exact paths. If those paths don't exist, the build fails.

Project Root Layout
directory tree
recipe-app-api/
├── app/                    # ALL Django code lives here — nothing outside
│   ├── manage.py           # Django management entry point
│   └── app/                # Inner app dir created by startproject
│       ├── settings.py
│       ├── urls.py
│       └── wsgi.py
│
├── .github/workflows/
│   └── checks.yml          # GitHub Actions: auto-test on every push
│
├── Dockerfile              # Blueprint for the container image
├── docker-compose.yml      # Dev orchestration — how to run locally
├── .dockerignore           # Exclude files from the build context
├── requirements.txt        # Production Python packages
└── requirements.dev.txt    # Dev-only packages (testing, linting)

Why app/ is a separate directory

The app/ directory is the boundary between your code and Docker's build context. The Dockerfile copies only this directory into the image: COPY ./app /app. This means files at the project root — your README.md, your CI config, your local notes — never end up in the production image. Smaller images. Cleaner separation.

🚨
Critical: app/ must exist before building

The COPY ./app /app instruction in the Dockerfile will fail with a build error if the app/ directory is missing — even if it is empty. Create it immediately when setting up a new project: mkdir app. You encountered this exact error today.

04
Dockerfile Deep-Dive
Every instruction explained — the what, the why, and the how

A Dockerfile is a sequence of instructions Docker executes top-to-bottom to build an image. Each instruction creates a new read-only layer. Layers are cached independently — understanding this cache is the difference between 2-second rebuilds and 3-minute rebuilds.

Core Concept

Docker Layer Caching

Docker reuses a cached layer if nothing above it has changed. Once a layer is invalidated (e.g. a file it depends on changed), all subsequent layers are also invalidated and re-executed. This is why the order of instructions in a Dockerfile matters enormously.

Rule of thumb: put things that change least frequently at the top, things that change most frequently at the bottom. Dependencies change less often than source code, so always copy and install requirements before copying your app code.

Dockerfile
dockerfile
FROM python:3.9-alpine3.13
LABEL maintainer="CODEANDCORE.COM"

ENV PYTHONUNBUFFERED=1

COPY ./requirements.txt /tmp/requirements.txt
COPY ./requirements.dev.txt /tmp/requirements.dev.txt
COPY ./app /app

WORKDIR /app
EXPOSE 8000

ARG Dev=false

RUN python -m venv /py && \
    /py/bin/pip install --upgrade pip && \
    /py/bin/pip install -r /tmp/requirements.txt && \
    if [ "$Dev" = "true" ]; then \
        /py/bin/pip install -r /tmp/requirements.dev.txt; \
    fi && \
    rm -rf /tmp && \
    adduser \
        --disabled-password \
        --no-create-home \
        django-user

ENV PATH="/py/bin:$PATH"

USER django-user

Line-by-line explanation

1
FROM python:3.9-alpine3.13
This is the base image — the starting point for your own image. We are using the official Python 3.9 image built on top of Alpine Linux 3.13. Every subsequent instruction adds layers on top of this.
Why Alpine? The standard python:3.9 image is ~900MB. The Alpine variant is ~45MB. Smaller images pull faster in CI, deploy faster, and have a smaller attack surface. Alpine is a minimal, security-focused Linux distribution — perfect for production containers.
2
ENV PYTHONUNBUFFERED=1
By default, Python buffers its output — it holds log messages in memory and flushes them in batches. Inside Docker, this means your application logs may never appear in docker compose logs if the process crashes.
Why this matters: Setting PYTHONUNBUFFERED=1 forces Python to write directly to stdout/stderr without buffering. Every print statement and log message appears immediately in the container logs. This is critical for debugging in production.
3
COPY requirements before COPY app
We copy requirements.txt and requirements.dev.txt before copying ./app. This is intentional layer caching optimisation.
The cache trick: When you change a Python file in app/, Docker invalidates the COPY ./app layer and everything below it. But the COPY ./requirements.txt layer above it is unchanged, so the pip install step is still cached. Without this ordering, every code change would reinstall all packages — turning a 3-second rebuild into a 3-minute one.
4
ARG Dev=false
ARG declares a build-time variable. When you run docker compose build locally, Compose passes Dev=true via the args block in docker-compose.yml. In CI and production, it defaults to false.
Why separate dev dependencies? Tools like flake8 (linter) and test runners are only needed during development and CI. Including them in production images adds unnecessary size and potential vulnerabilities. The if [ "$Dev" = "true" ] shell conditional inside the RUN instruction installs them only when explicitly requested.
5
python -m venv /py — Virtual Environment Inside Docker
We create a virtual environment at /py even though we are already inside a container. Then we install all packages into /py/bin/pip rather than the system pip.
Why a venv inside a container? Alpine Linux has its own Python installation used by system tools. Installing packages with system pip risks overwriting system Python packages and breaking Alpine internals. The virtual environment keeps your application packages completely isolated from the OS. It also makes the setup explicit and reproducible.
6
rm -rf /tmp — Clean Up After Installing
After pip installs the packages from /tmp/requirements.txt, we delete the entire /tmp directory in the same RUN instruction.
Why in the same RUN? Each RUN creates a new layer. If you deleted /tmp in a separate RUN instruction, the files would still exist in the previous layer and be baked into the image. Doing the cleanup in the same RUN means the files never persist to any layer, keeping the image as small as possible.
7
adduser django-user — The Principle of Least Privilege
By default, Docker containers run as root. We create a dedicated django-user with no password (--disabled-password) and no home directory (--no-create-home), then switch to it with USER django-user.
Security: If an attacker exploits a vulnerability in your Django application, the process running as root would have full control over the container and potentially the host. Running as a non-root user limits the blast radius of any security breach. This is a required practice in any production container setup.
8
ENV PATH="/py/bin:$PATH"
This adds the virtual environment's bin directory to the system PATH. After this line, running python or django-admin in the container automatically uses the venv versions.
Why this must come after RUN: The PATH ENV is set after the RUN block so that the venv is fully created before we try to use it. The ENV persists into the running container, so every subsequent docker compose run command uses the venv automatically.
05
Requirements Files
Pinning dependencies for reproducible, safe builds

Python package management is one of the most common sources of broken builds. A package that works today may release a breaking change tomorrow. The solution is version pinning — specifying exactly which versions your application requires.

📦

requirements.txt

Production dependencies only. Installed in every build. Kept as lean as possible — every unnecessary package is a potential security vulnerability.

🔧

requirements.dev.txt

Development and CI tools only. Installed when Dev=true. Contains linters, test runners, and debug tools that must never reach production.

requirements.txt — Production
pip
# Production dependencies
# Lower bound: minimum version with features we need
# Upper bound: prevents accidental breaking upgrades
Django>=3.2.3,<4.0
djangorestframework>=3.0,<4.0
requirements.dev.txt — Development & CI only
pip
# Dev & CI tools — never installed in production
flake8>=3.9.2,<4.0.0   # PEP 8 linter
Version Ranges Explained

Why >=3.2.3,<4.0 instead of ==3.2.3?

Pinning to an exact version (==3.2.3) is overly strict. It prevents you from getting security patches in 3.2.4, 3.2.5, etc.

The range >=3.2.3,<4.0 uses semantic versioning: allow any patch release (bug fixes, security patches) and any minor release within the major version, but block the next major version which may contain breaking changes. This is the professional standard for production dependency management.

⚠️
After changing requirements

Run docker compose build --no-cache to force a complete rebuild. The --no-cache flag ignores all cached layers, ensuring pip fetches the latest versions within your specified ranges. Without it, Docker may use a stale cached pip install layer.

06
Docker Compose
Orchestrating your development environment with a single command

docker compose is the tool for defining and running multi-service Docker applications. Even with a single service, it is the right choice — it stores all your run configuration (ports, volumes, commands, build arguments) in a version-controlled YAML file so you never have to remember long docker run flags.

Core Concept

Compose V2 vs the old docker-compose

There are two generations of Docker Compose. The original standalone binary was called docker-compose (with a hyphen). It was written in Python and shipped separately. Docker Compose V2 is the modern rewrite in Go, built directly into the Docker CLI as a plugin: docker compose (with a space).

On Ubuntu 24.04 (used by GitHub Actions), only V2 is pre-installed. Using docker-compose in your CI workflow will fail with "command not found". Always use docker compose (space) in all scripts, Makefiles, and CI configuration files.

docker-compose.yml
yaml
# No 'version:' key — it is obsolete in Compose V2 and causes warnings
services:
  app:
    build:
      context: .            # Use current directory as build context
      args:
        - Dev=true          # Pass ARG to Dockerfile — installs dev deps
    ports:
      - "8000:8000"          # host_port:container_port
    volumes:
      - ./app:/app          # Mount local code into container
    command: >
      sh -c "python manage.py migrate &&
             python manage.py runserver 0.0.0.0:8000"

Key directives explained

A
build.context: .
Tells Compose where the build context is — the directory whose contents are sent to the Docker daemon. The dot means the current directory (your project root). This is also where Docker looks for the Dockerfile.
B
volumes: ./app:/app
Mounts your local ./app directory into the container at /app. Any change you make to a file in ./app on your host immediately appears inside the running container — no rebuild required. This is how live code reloading works with Django's development server.
Important: Volumes only apply at runtime (docker compose up). They do not affect the built image. The image always uses the code copied during docker compose build. In production you would not use a volume — the code is baked into the image itself.
C
command: sh -c "migrate && runserver"
This overrides the default CMD in the Dockerfile. On every docker compose up, the container runs database migrations first (applying any pending schema changes) and then starts the Django development server.
Why migrate on startup? In development, you frequently add models and create migrations. Having migrate run automatically means the database schema is always up to date when you start the server. In production you would run migrations as a separate step before deploying.
07
.dockerignore
Controlling what enters the build context

When you run docker build . or docker compose build, Docker sends the entire project directory to the Docker daemon as the build context. This happens before any COPY instruction runs. A large build context means slow builds even if you only COPY a small subset of files.

Core Concept

The Build Context Transfer

When you run docker build ., the first thing Docker prints is something like "Sending build context to Docker daemon 4.5MB". That is every file in your project folder being streamed to the Docker daemon over a socket. Without a .dockerignore, this includes your .git/ folder, test fixtures, local database files, and everything else.

The .dockerignore file works exactly like .gitignore — it tells Docker which files and directories to exclude from the build context. Less data transferred, faster builds, and no accidental secrets baked into images.

.dockerignore
config
# Version control — never needed in the image
.git
.gitignore

# Documentation — excluded from the image
*.md
index.html

# Python cache files — rebuilt automatically inside the container
__pycache__
*.pyc
*.pyo
.pytest_cache

# Local environment files — NEVER bake secrets into an image
.env
.env.*
*.env

# Editor configuration
.vscode
.idea

# OS artifacts
.DS_Store
Thumbs.db
🚨
Critical: The dot prefix is not optional

The file must be named .dockerignore with a leading dot. A file named dockerignore (no dot) is silently ignored by Docker — your entire project directory will be sent as the build context and no files will be excluded. This was one of the issues found in your project today.

🔒
Security: Never include .env files in images

If your .env file contains database passwords, API keys, or Django's SECRET_KEY and it is copied into the image, those secrets become visible to anyone who can pull the image from your registry. Always list .env and .env.* in your .dockerignore.

08
Django Project Setup
Bootstrapping Django entirely inside the container

The golden rule of Docker-based development is: run every application command inside the container, never on your host machine. This means Django project creation, management commands, tests, and migrations all happen via docker compose run.

📚 Mental Model

Think of docker compose run --rm app sh -c "command" as "SSH into a temporary copy of the container, run this one command, then destroy the container." The --rm flag means the container is deleted after the command finishes. Your code changes persist because the app/ directory is mounted as a volume.

1
Build the image first
Before any other command, you must build the Docker image. This executes every instruction in your Dockerfile: pulls the base image, installs Python packages, creates the user. Repeat this whenever you change the Dockerfile or requirements files.
Terminal
shell
docker compose build
2
Create the Django project inside the container
Run startproject inside the container. The trailing dot (.) tells Django to create the project in the current directory (/app inside the container, which is mapped to your local ./app) rather than creating a nested subfolder.
Terminal
shell
docker compose run --rm app sh -c "django-admin startproject app ."
Why does this create files on your host? The ./app:/app volume mount works both ways. Files created inside the container at /app appear immediately in your local ./app directory. After this command you will see manage.py and the inner app/ folder appear in VS Code.
3
Start the development server
Compose runs your command from docker-compose.yml: first migrates the database, then starts Django on all network interfaces (0.0.0.0) so the port is accessible from your host at http://localhost:8000.
Terminal
shell
docker compose up
4
Run the test suite
Always run tests inside the container. The --rm flag destroys the container after tests complete, keeping your environment clean. The exit code from this command is used by CI to determine pass/fail.
Terminal
shell
docker compose run --rm app sh -c "python manage.py test"
5
Run the linter
Flake8 checks your code for PEP 8 style violations. It runs on the entire codebase (.). Configure exceptions and max line length in a .flake8 file at the project root.
Terminal
shell
docker compose run --rm app sh -c "flake8 ."
09
CI/CD with GitHub Actions
Automated testing and linting on every single commit

Continuous Integration (CI) means running your tests automatically every time code is pushed. If the tests fail, you know immediately — before bad code reaches your main branch or production. GitHub Actions is GitHub's built-in CI runner and it is completely free for public repositories.

Core Concept

How GitHub Actions Works

When you push a commit, GitHub reads the YAML files inside .github/workflows/ and triggers the matching workflows. Each workflow runs on a fresh virtual machine (called a runner) in GitHub's infrastructure. The runner executes your steps in order: checkout code, log into Docker Hub, build the image, run tests.

If any step exits with a non-zero code, the workflow fails and GitHub marks the commit with a red ✕. A green ✓ means all steps passed. You can require a green check before merging a pull request — this is the foundation of safe team development.

.github/workflows/checks.yml
yaml
---
name: Checks

on: [push]          # Trigger on every push to any branch

jobs:
  test-lint:
    name: Test and Lint
    runs-on: ubuntu-24.04   # Fresh Ubuntu VM for every run

    steps:
      # Step 1: Authenticate BEFORE checkout to avoid rate limits
      - name: Login to Docker Hub
        uses: docker/login-action@v3
        with:
          username: ${{ secrets.DOCKERHUB_USER }}
          password: ${{ secrets.DOCKERHUB_TOKEN }}

      # Step 2: Clone your repository onto the runner
      - name: Check out code
        uses: actions/checkout@v4

      # Step 3: Run Django tests inside the container
      - name: Test
        run: docker compose run --rm app sh -c "python manage.py test"

      # Step 4: Run PEP 8 linter inside the container
      - name: Lint
        run: docker compose run --rm app sh -c "flake8 ."

Why login before checkout?

At first glance this ordering seems wrong — why log into Docker Hub before you have the code? The answer is about when Docker pulls images. GitHub's hosted runners share IP addresses. An unauthenticated pull could hit Docker Hub's rate limit at any moment. Authenticating first ensures the entire workflow — including the base image pull during docker compose run — is authenticated.

Setting up GitHub Secrets

Secrets are encrypted environment variables stored in your GitHub repository. They are injected into workflow runs but never exposed in logs. Go to your repo on GitHub: Settings → Secrets and variables → Actions → New repository secret.

Secret NameWhat to put in itWhere to get it
DOCKERHUB_USER Your Docker Hub username (not email) hub.docker.com → Account Settings
DOCKERHUB_TOKEN A Docker Hub Access Token (not your password) hub.docker.com → Account Settings → Security → New Access Token
✅
Use Access Tokens, not passwords

Docker Hub Access Tokens are scoped credentials that can be revoked independently of your account password. If a token is accidentally exposed, you revoke it without changing your password. Create a dedicated token for each CI pipeline so you can audit and revoke access independently.

Troubleshooting Guide
Every error you will encounter and exactly how to fix it

These are real errors encountered while building this exact project. Each one has a root cause, a diagnostic command, and a verified fix.

!
the Dockerfile cannot be empty
The build context is sending a 0-byte file. The Dockerfile exists in your editor but has not been saved to disk yet.
Diagnose: Get-Item Dockerfile | Select-Object Length will show 0. Press Ctrl+S in VS Code to save the file.
!
COPY failed: file not found — ./app
The app/ directory does not exist in your project root. Docker cannot copy a directory that doesn't exist, even if the Dockerfile is valid.
Fix: Run mkdir app in the project root. The directory can be empty — Docker will copy an empty directory successfully.
!
docker-compose: command not found (in CI)
Your GitHub Actions workflow uses docker-compose (with a hyphen). Ubuntu 24.04 runners only have Docker Compose V2 installed as docker compose (with a space).
Fix: Replace all docker-compose with docker compose in your checks.yml file. This is already fixed in this project.
!
the attribute `version` is obsolete
Your docker-compose.yml has version: '3.9' at the top. Compose V2 no longer uses this field and prints a warning on every command.
Fix: Delete the version: line entirely. Compose V2 auto-detects the format. This is already fixed in this project.
!
.dockerignore has no effect / entire project in context
The ignore file is named dockerignore without a leading dot. Docker silently ignores files without the dot prefix.
Fix: Rename to .dockerignore. In PowerShell: Rename-Item dockerignore .dockerignore. This was fixed in your project today.
!
toomanyrequests: Docker Hub rate limit exceeded
Your CI pipeline is pulling Docker images without authenticating. GitHub-hosted runners share IP addresses with other users and collectively exhaust the 100 unauthenticated pulls per 6 hours.
Fix: Ensure docker/login-action runs as the first step in your workflow, before actions/checkout. Verify DOCKERHUB_USER and DOCKERHUB_TOKEN secrets are set in your repo settings.
