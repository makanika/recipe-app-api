What is MQTT?

Message Queuing Telemetry Transport — the language of machines

MQTT stands for Message Queuing Telemetry Transport. It is a lightweight, open network protocol designed specifically for sending small pieces of data between many devices in situations where bandwidth is tight, battery life matters, or network reliability cannot be guaranteed. It was invented by Andy Stanford-Clark of IBM and Arlen Nipper of Arcom in 1999 to monitor oil pipelines over satellite connections, where every byte and every dropped connection cost real money.

Today MQTT is the default language of the Internet of Things. It runs on thermostats, generators, water pumps, hotel room controllers, fire panels, solar inverters, and medical devices. It is what your ESP32 and Raspberry Pi Pico W should speak as soon as they need to report anything to the outside world.

The reason MQTT became dominant is not by accident. The entire protocol was built around a single radical constraint: the device you are talking to might disconnect at any moment without warning, and the network connecting it might be slow, expensive, and unreliable. When you design for the worst case, everything easier becomes trivially simple. MQTT brings that discipline to every project that uses it.

MQTT versus HTTP — Why the Difference Matters

Most developers first encounter data transfer through HTTP, the same protocol used by web browsers. HTTP works on a request-response model: the client asks a question, the server answers, and the connection closes. This is excellent for browsing a web page or calling a REST API.

But imagine a temperature sensor that needs to report its reading every five seconds. Using HTTP, the device would need to open a TCP connection, send a header-heavy request, wait for the server response, and close the connection — five hundred times per hour. The overhead of the protocol massively outweighs the payload. On a battery-powered device this is a disaster. On a congested network this is reckless.

MQTT flips the model. The device opens a single long-lived connection to a central server called the broker, and that connection stays open. Sending a temperature reading is then just a tiny publish message — as small as two bytes of overhead plus your payload. The broker keeps track of every connected device, fans messages out to subscribers, and stores state on behalf of devices that go offline.

Key fact

The minimum MQTT packet overhead is 2 bytes. The minimum HTTP request overhead is several hundred bytes. For a sensor sending small payloads thousands of times a day, this difference is enormous in battery life and bandwidth cost.

MQTT is Ideal For

Sensors, microcontrollers, mobile devices, poor networks, high publish frequency, many devices, event-driven telemetry.

HTTP is Better For

Human web browsing, stateless REST APIs, file transfers, one-time requests, situations where caching and CDN matter.

How MQTT Works

Publish, Subscribe, and the three-actor model

MQTT is built around three actors: the publisher, the broker, and the subscriber. Understanding the role of each one is the foundation of every system you will ever build with this protocol.

Publisher
(ESP32 / Pico)
sends data
→
Broker
(Mosquitto)
routes messages
→
Subscriber
(Python Bridge)
receives & stores
→
PostgreSQL
(Database)
persists data
→
Web Page
(Dashboard)
displays data

The Publisher

A publisher is any client that sends a message to the broker. This is almost always a device, a sensor, or a piece of software that knows something useful. The publisher does not know who will receive its message. It does not care. It simply says: "I am sending a temperature of 22.4 degrees to the topic building/floor2/room12/temperature" and then goes back to measuring. This decoupling is powerful because you can add subscribers without touching the device firmware at all.

The Broker

The broker is the central server. It receives messages from publishers, inspects their topics, and delivers them to every subscriber that has registered interest in that topic. It also manages connections, handles authentication, persists retained messages, and enforces Quality of Service guarantees. You will almost always use Mosquitto or EMQX as your broker. In this book we use Mosquitto running in Docker.

The Subscriber

A subscriber is any client that has told the broker "I want to receive messages for this topic." There can be zero, one, or a thousand subscribers for any given topic. In our system the most important subscriber is the bridge service — a Python script that receives every telemetry message and inserts it into PostgreSQL.

The Publish-Subscribe Pattern

The technical name for this pattern is pub/sub. The key property is decoupling. Publishers and subscribers never know about each other directly. They are connected only through topic names. This means you can change your storage backend, add a new analytics subscriber, or replace your device firmware without breaking the rest of the system. Every serious IoT architecture uses this pattern for exactly that reason.

Design principle

In pub/sub architecture, your devices are free from knowing where their data goes. The broker owns routing. This is what makes your system extensible without firmware changes.

The Broker — Mosquitto in Docker

Installing, configuring, and running the message router

Mosquitto is the most widely deployed open-source MQTT broker in the world. It is small, fast, and production-proven. We run it in Docker so the installation is identical on every machine and every server, and so it integrates cleanly with Postgres, our API, and our bridge service in the same Docker network.

Mosquitto configuration file

This is the minimal configuration for a development broker. It enables anonymous connections on port 1883 and WebSocket connections on port 9001 so your web dashboard can connect directly.

conf
# mosquitto/config/mosquitto.conf

# Standard MQTT port
listener 1883
protocol mqtt

# WebSocket port (browsers use this)
listener 9001
protocol websockets

# Allow unauthenticated connections in development
allow_anonymous true

# Persist messages so retained topics survive restarts
persistence true
persistence_location /mosquitto/data/

# Log everything to stdout and file
log_dest stdout
log_dest file /mosquitto/log/mosquitto.log
log_type all
Production note

In production, set allow_anonymous false and create a password file with mosquitto_passwd. Never expose the broker directly to the internet without authentication and TLS.

Testing the broker from your terminal

Terminal 2 — publish a test message:

  -t "building/floor1/room101/temperature" \
  -m '{"value": 22.4, "unit": "celsius"}'

You should immediately see the message appear in Terminal 1. If you do, your broker is working correctly.

04

Topics and Quality of Service

The addressing system and delivery guarantees

Topics — the address of every message

A topic is a UTF-8 string that acts as the address and routing key for an MQTT message. Topics are structured with forward slashes as separators, like a file path or a URL, but they are entirely arbitrary — you define the naming convention for your system. Good topic design is one of the most important architecture decisions you will make.

Recommended topic schema

text
{site}/{building}/{floor}/{room}/{device-type}/{metric}

Examples:
  nairobi/main/floor3/room310/temperature/celsius
  daresalaam/onomo/floor1/room112/humidity/percent
  kampala/hotel-a/basement/generator/fuel/liters
  nairobi/main/roof/fire-panel/zone3/alarm

The most important rule is consistency. If every topic follows the same depth and structure, you can use wildcards to subscribe to logical groupings and your bridge code stays simple. If topics are inconsistent you will write special cases forever.

Wildcards

MQTT defines two wildcard characters that subscribers can use to match multiple topics.

WildcardNameBehaviourExample subscription
+Single-levelMatches exactly one levelbuilding/+/room101/temperature
#Multi-levelMatches everything from here downbuilding/floor2/#

Our bridge service subscribes to # — all topics — and records the full topic path as a column in Postgres. This means adding new sensors requires zero changes to the bridge code.

Quality of Service (QoS)

QoS is the guarantee that MQTT makes about whether a message is delivered. There are three levels, and choosing the wrong one causes either lost data or wasted processing. You must understand all three to design reliable systems.

LevelNameGuaranteeUse when
0 At most once Fire and forget. Message may be lost. High-frequency sensor data where occasional loss is acceptable.
1 At least once Delivered at least once. May arrive twice. Telemetry data where you need every reading but can handle duplicates.
2 Exactly once Delivered exactly once. Highest overhead. Commands, financial events, actions that must not be duplicated.
Practical guidance

Use QoS 1 for telemetry going into Postgres. The bridge should make Postgres inserts idempotent so duplicates from QoS 1 retries do not create corrupted data. Use QoS 2 for control commands that switch relays or execute actions with side effects.

Retained Messages

When a publisher sets the retain flag, the broker stores the last message on that topic and delivers it immediately to any new subscriber. This is extraordinarily useful for device state: if your dashboard connects to the broker an hour after the generator reported its fuel level, it immediately receives the last known value rather than waiting for the next publish.

shell
# Publish with retain flag
mosquitto_pub -h localhost -p 1883 \
  -t "generator/fuel" \
  -m '{"level_liters": 450, "capacity": 600}' \
  --retain
05

Why PostgreSQL?

The right database for time-series telemetry and operational data

When MQTT messages arrive at the broker they exist in memory. The broker does not persist your telemetry history — that is not its job. You need a proper database to store readings durably, query them efficiently, build reports on them, and serve them to web dashboards. PostgreSQL is the right choice for this and the reasons are worth understanding clearly.

MQTT produces time-series data: rows of observations with a timestamp, a device identifier, a metric name, and a value. Every sensor you add simply adds more rows to the same table structure. Postgres handles this naturally and scales far further than most IoT deployments ever need. Pure time-series databases like InfluxDB exist and are worth knowing about, but Postgres lets you join your telemetry data with your device registry, your alert history, your user accounts, and your business logic in a single query language you already know.

Strong Points for This Use Case

ACID transactions, JSONB column type for flexible payloads, partial indexes for performance, pg_partman for time-based partitioning, and PostGIS for location data.

What to Watch

High-frequency inserts need proper indexing strategy. Partitioning by month prevents table bloat. Vacuuming and connection pooling are important at scale.

JSONB — Your Bridge to Flexible Payloads

MQTT messages arrive as raw bytes. You will typically send JSON payloads because they are human-readable and easy to extend without changing your schema. PostgreSQL's JSONB column type stores JSON as a binary-indexed format that supports operators for querying inside the document without parsing it in application code.

The best design pattern is a hybrid schema: keep your known, indexed columns as proper SQL types (timestamp, device_id, topic) and put the full raw payload in a JSONB column. This gives you efficient indexed queries on the common filters while keeping the raw data fully accessible.

06

Schema Design

Designing a Postgres schema that grows cleanly with your devices

Good schema design is the difference between a database that stays fast and queryable for years and one that becomes a pain within months. This schema is battle-tested for IoT telemetry.

The devices table — your registry

Every device that publishes MQTT messages should have a record in the devices table. This lets you join telemetry to metadata like location, hardware type, and install date without repeating that information in every row.

sql
-- db/init.sql

CREATE TABLE devices (
    id          SERIAL PRIMARY KEY,
    device_id   TEXT NOT NULL UNIQUE,
    site        TEXT NOT NULL,
    building    TEXT,
    floor       TEXT,
    room        TEXT,
    device_type TEXT NOT NULL,
    description TEXT,
    installed   DATE,
    is_active   BOOLEAN NOT NULL DEFAULT true,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- Index for the most common lookup
CREATE INDEX idx_devices_device_id ON devices (device_id);
CREATE INDEX idx_devices_site      ON devices (site);

The telemetry table — your event log

Every MQTT message that arrives becomes one row in this table. Append-only, never update. This preserves the complete history and allows accurate replay and audit.

sql
CREATE TABLE telemetry (
    id          BIGSERIAL PRIMARY KEY,
    received_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    device_id   TEXT NOT NULL,
    topic       TEXT NOT NULL,
    qos         SMALLINT NOT NULL DEFAULT 0,
    payload     JSONB NOT NULL,
    -- Extracted common fields for indexed queries
    metric      TEXT,
    value_num   DOUBLE PRECISION,
    unit        TEXT
);

-- Performance indexes
CREATE INDEX idx_telemetry_received_at ON telemetry (received_at DESC);
CREATE INDEX idx_telemetry_device_id   ON telemetry (device_id);
CREATE INDEX idx_telemetry_topic       ON telemetry (topic);
CREATE INDEX idx_telemetry_metric      ON telemetry (metric);

-- JSONB index for payload queries
CREATE INDEX idx_telemetry_payload ON telemetry USING gin (payload);

The device_state table — last known values

The telemetry table tells you everything that happened. But to show a live dashboard you need to know the current value for each device without scanning millions of rows. The device_state table is an upsert target that stores only the latest reading per device and metric.

sql
CREATE TABLE device_state (
    device_id   TEXT NOT NULL,
    metric      TEXT NOT NULL,
    value_num   DOUBLE PRECISION,
    unit        TEXT,
    payload     JSONB,
    last_seen   TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    PRIMARY KEY (device_id, metric)
);

CREATE INDEX idx_device_state_last_seen ON device_state (last_seen DESC);

The alerts table — event escalation log

sql
CREATE TABLE alerts (
    id          BIGSERIAL PRIMARY KEY,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    device_id   TEXT NOT NULL,
    topic       TEXT NOT NULL,
    alert_type  TEXT NOT NULL,
    severity    TEXT NOT NULL CHECK (severity IN ('info', 'warning', 'critical')),
    message     TEXT NOT NULL,
    payload     JSONB,
    resolved_at TIMESTAMPTZ,
    resolved_by TEXT
);

CREATE INDEX idx_alerts_device_id  ON alerts (device_id);
CREATE INDEX idx_alerts_created_at ON alerts (created_at DESC);
CREATE INDEX idx_alerts_severity   ON alerts (severity)
    WHERE resolved_at IS NULL;
Schema rule

Never delete rows from telemetry or alerts. These are immutable event logs. Add a resolved_at column to alerts and a superseded_by column to state if you need logical deletion. Your future self and your auditors will thank you.

Writing the Bridge Service

The Python service that connects MQTT to PostgreSQL

The bridge is the most important service in our stack. It subscribes to the MQTT broker, receives every message, validates and parses the payload, and writes it to Postgres. It must be resilient to broker disconnections, handle malformed payloads gracefully, and run continuously without leaking memory.

Dependencies

text
# bridge/requirements.txt
paho-mqtt==2.1.0
psycopg2-binary==2.9.9
python-dotenv==1.0.1

The complete bridge service

python
# bridge/bridge.py
"""
MQTT → PostgreSQL bridge service.
Subscribes to all topics (#) on the broker and persists
every message to the telemetry and device_state tables.
"""

import json
import logging
import os
import time
from datetime import datetime, timezone

import paho.mqtt.client as mqtt
import psycopg2
from psycopg2.extras import Json
from dotenv import load_dotenv

load_dotenv()

# ── Logging ───────────────────────────────────────────────
logging.basicConfig(
    level=logging.INFO,
    format="%(asctime)s [%(levelname)s] %(message)s",
    datefmt="%Y-%m-%d %H:%M:%S",
)
log = logging.getLogger(__name__)

# ── Config ────────────────────────────────────────────────
MQTT_HOST     = os.getenv("MQTT_HOST", "localhost")
MQTT_PORT     = int(os.getenv("MQTT_PORT", 1883))
MQTT_TOPIC    = os.getenv("MQTT_TOPIC", "#")
MQTT_CLIENT_ID = "postgres-bridge-01"

DB_DSN = (
    f"host={os.getenv('DB_HOST', 'localhost')} "
    f"port={os.getenv('DB_PORT', '5432')} "
    f"dbname={os.getenv('DB_NAME', 'iot')} "
    f"user={os.getenv('DB_USER', 'iot')} "
    f"password={os.getenv('DB_PASSWORD', 'iot')}"
)

# ── Database connection ───────────────────────────────────
def get_db_connection():
    """Create a database connection with retry logic."""
    for attempt in range(10):
        try:
            conn = psycopg2.connect(DB_DSN)
            conn.autocommit = False
            log.info("Database connected.")
            return conn
        except psycopg2.OperationalError as e:
            log.warning(f"DB not ready (attempt {attempt+1}/10): {e}")
            time.sleep(3)
    raise RuntimeError("Could not connect to database after 10 attempts.")


# ── Payload parsing ───────────────────────────────────────
def parse_payload(raw: bytes) -> dict:
    """Parse raw MQTT bytes as JSON. Return empty dict on failure."""
    try:
        return json.loads(raw.decode("utf-8"))
    except (json.JSONDecodeError, UnicodeDecodeError) as e:
        log.warning(f"Invalid payload: {e} — raw: {raw[:100]}")
        return {"_raw": raw.decode("utf-8", errors="replace")}


def extract_numeric(payload: dict) -> tuple[float | None, str | None]:
    """Try to extract a numeric value and unit from common payload shapes."""
    for key in ("value", "val", "reading", "level", "temp", "temperature"):
        if key in payload:
            try:
                return float(payload[key]), payload.get("unit")
            except (TypeError, ValueError):
                pass
    return None, None


# ── Database writes ───────────────────────────────────────
def insert_telemetry(cursor, topic: str, qos: int, payload: dict):
    """Insert one message into the telemetry event log."""
    value_num, unit = extract_numeric(payload)
    # Extract metric from last topic segment
    metric = topic.split("/")[-1] if "/" in topic else topic

    cursor.execute(
        """
        INSERT INTO telemetry
            (topic, device_id, qos, payload, metric, value_num, unit)
        VALUES (%s, %s, %s, %s, %s, %s, %s)
        """,
        (
            topic,
            payload.get("device_id", topic.split("/")[0]),
            qos,
            Json(payload),
            metric,
            value_num,
            unit,
        ),
    )


def upsert_device_state(cursor, topic: str, payload: dict):
    """Update the current state table with the latest value."""
    value_num, unit = extract_numeric(payload)
    metric = topic.split("/")[-1] if "/" in topic else topic
    device_id = payload.get("device_id", topic.split("/")[0])

    cursor.execute(
        """
        INSERT INTO device_state (device_id, metric, value_num, unit, payload, last_seen)
        VALUES (%s, %s, %s, %s, %s, NOW())
        ON CONFLICT (device_id, metric) DO UPDATE
            SET value_num = EXCLUDED.value_num,
                unit      = EXCLUDED.unit,
                payload   = EXCLUDED.payload,
                last_seen = EXCLUDED.last_seen
        """,
        (device_id, metric, value_num, unit, Json(payload)),
    )


# ── MQTT callbacks ────────────────────────────────────────
def on_connect(client, userdata, flags, reason_code, properties=None):
    if reason_code == 0:
        log.info(f"Connected to broker {MQTT_HOST}:{MQTT_PORT}")
        client.subscribe(MQTT_TOPIC, qos=1)
        log.info(f"Subscribed to: {MQTT_TOPIC}")
    else:
        log.error(f"Connection refused, reason code: {reason_code}")


def on_disconnect(client, userdata, disconnect_flags, reason_code, properties=None):
    log.warning(f"Disconnected from broker (rc={reason_code}). Reconnecting...")


def make_on_message(db_conn):
    """Factory: returns an on_message callback with the db connection in closure."""

    def on_message(client, userdata, msg):
        topic   = msg.topic
        qos     = msg.qos
        payload = parse_payload(msg.payload)

        log.debug(f"RX {topic} qos={qos} payload={payload}")

        try:
            with db_conn.cursor() as cursor:
                insert_telemetry(cursor, topic, qos, payload)
                upsert_device_state(cursor, topic, payload)
            db_conn.commit()
        except psycopg2.Error as e:
            log.error(f"DB write failed for topic {topic}: {e}")
            db_conn.rollback()

    return on_message


# ── Entry point ───────────────────────────────────────────
def main():
    log.info("Starting MQTT → PostgreSQL bridge...")

    db_conn = get_db_connection()

    client = mqtt.Client(
        client_id=MQTT_CLIENT_ID,
        protocol=mqtt.MQTTv5,
    )
    client.on_connect    = on_connect
    client.on_disconnect = on_disconnect
    client.on_message    = make_on_message(db_conn)

    client.connect(MQTT_HOST, MQTT_PORT, keepalive=60)

    try:
        client.loop_forever(retry_first_connection=True)
    except KeyboardInterrupt:
        log.info("Bridge shutting down.")
    finally:
        client.disconnect()
        db_conn.close()


if __name__ == "__main__":
    main()
Why loop_forever?

loop_forever() blocks the main thread and runs the MQTT network loop in the background, handling reconnections automatically. It is the correct pattern for a long-running bridge service. retry_first_connection=True means it will wait patiently if the broker starts after the bridge.

Worked Example — Temperature Logger

Simulating a hotel room sensor, end to end

We now have a running stack. Let us simulate a temperature sensor in hotel room 101 and verify the entire journey from device to Postgres.

Python sensor simulator

This script simulates a room sensor publishing readings every 10 seconds.

python
# simulator/temp_sensor.py
"""Simulates a temperature and humidity sensor in hotel room 101."""

import json
import random
import time
import paho.mqtt.client as mqtt

BROKER     = "localhost"
DEVICE_ID  = "temp-room101"
ROOM_TOPIC = "daresalaam/onomo/floor1/room101"

def publish_readings(client):
    temperature = round(random.uniform(20.0, 27.0), 2)
    humidity    = round(random.uniform(40.0, 65.0), 1)

    # Publish temperature
    client.publish(
        f"{ROOM_TOPIC}/temperature",
        json.dumps({
            "device_id": DEVICE_ID,
            "value": temperature,
            "unit": "celsius",
            "ts": int(time.time()),
        }),
        qos=1,
    )

    # Publish humidity
    client.publish(
        f"{ROOM_TOPIC}/humidity",
        json.dumps({
            "device_id": DEVICE_ID,
            "value": humidity,
            "unit": "percent",
            "ts": int(time.time()),
        }),
        qos=1,
    )
    print(f"Published: temp={temperature}°C  humidity={humidity}%")


def main():
    client = mqtt.Client(client_id="sim-room101", protocol=mqtt.MQTTv5)
    client.connect(BROKER, 1883, keepalive=60)
    client.loop_start()

    print("Sensor simulator started. Ctrl+C to stop.")
    try:
        while True:
            publish_readings(client)
            time.sleep(10)
    except KeyboardInterrupt:
        print("\nStopping simulator.")
    finally:
        client.loop_stop()
        client.disconnect()


if __name__ == "__main__":
    main()

Worked Example — Device Heartbeats

Tracking which devices are online and detecting silent failures

A heartbeat is a regular "I am alive" message that each device publishes on a known schedule. If the heartbeat stops arriving, the device has failed, lost power, or lost network. Without heartbeats you will not know a sensor has died until a human notices the dashboard shows stale data.

Adding heartbeat to firmware (MicroPython — Pico W)

python

    wlan = network.WLAN(network.STA_IF)
    wlan.active(True)
    wlan.connect(WIFI_SSID, WIFI_PASSWORD)
    for _ in range(20):
        if wlan.isconnected():
            print("WiFi connected:", wlan.ifconfig())
            return True
        time.sleep(1)
    return False


def main():

        print("WiFi failed — rebooting")
        import machine; machine.reset()

    client = MQTTClient(
        DEVICE_ID,
        BROKER_IP,
        port=1883,
        keepalive=60,
    )
    client.connect()
    print("MQTT connected")

    heartbeat_topic = f"devices/{DEVICE_ID}/heartbeat"

    while True:
        payload = json.dumps({
            "device_id": DEVICE_ID,
            "status": "online",
            "uptime_s": time.ticks_ms() // 1000,
            "ts": time.time(),

Worked Example — Command and Control

Sending commands from the server to devices over MQTT

MQTT is bidirectional. The same broker that carries telemetry up from devices to the server can carry commands down from the server to devices. This is the foundation of every control system: generator start/stop, room thermostat setpoints, announcement triggers, and relay switching.

Command topic convention

text
Telemetry (device → server):   devices/{device_id}/telemetry/{metric}
Commands  (server → device):   devices/{device_id}/cmd/{action}
Responses (device → server):   devices/{device_id}/rsp/{action}

Postgres: commands table

sql
CREATE TABLE commands (
    id          BIGSERIAL PRIMARY KEY,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    device_id   TEXT NOT NULL,
    action      TEXT NOT NULL,
    payload     JSONB NOT NULL DEFAULT '{}',
    sent_at     TIMESTAMPTZ,
    acked_at    TIMESTAMPTZ,
    status      TEXT NOT NULL DEFAULT 'pending'
                CHECK (status IN ('pending','sent','acked','failed')),
    issued_by   TEXT
);

CREATE INDEX idx_commands_device_id ON commands (device_id);
CREATE INDEX idx_commands_status    ON commands (status)
    WHERE status IN ('pending', 'sent');

Command publisher (Python)

python
# Send a command to a device and record it in Postgres
import json, time
import paho.mqtt.client as mqtt
import psycopg2
from psycopg2.extras import Json

def send_command(device_id: str, action: str, payload: dict):
    """Publish a command to a device and record it in the database."""

    db = psycopg2.connect(DB_DSN)
    topic = f"devices/{device_id}/cmd/{action}"
    full_payload = {
        "action":    action,
        "device_id": device_id,
        "payload":   payload,
        "ts":        int(time.time()),
    }

    # Record in DB first (QoS 2 ensures delivery)
    with db.cursor() as cur:
        cur.execute(
            """
            INSERT INTO commands (device_id, action, payload, sent_at, status)
            VALUES (%s, %s, %s, NOW(), 'sent')
            RETURNING id
            """,
            (device_id, action, Json(full_payload)),
        )
        cmd_id = cur.fetchone()[0]
    db.commit()

    # Publish with QoS 2 (exactly once — important for control actions)
    client = mqtt.Client(protocol=mqtt.MQTTv5)
    client.connect("localhost", 1883)
    result = client.publish(topic, json.dumps(full_payload), qos=2)
    result.wait_for_publish(timeout=5)
    client.disconnect()
    db.close()

    print(f"Command {cmd_id} sent: {action} → {device_id}")
    return cmd_id

Worked Example — Alert Engine

Detecting out-of-range conditions and escalating them

Raw telemetry stored in Postgres is useful for reports and dashboards, but the real value of a connected system is automatic detection of problems. The alert engine reads the telemetry stream, evaluates each reading against configured thresholds, and writes an alert record when a threshold is breached.

Alert rules table

sql
CREATE TABLE alert_rules (
    id          SERIAL PRIMARY KEY,
    device_id   TEXT,          -- NULL means apply to all devices
    metric      TEXT NOT NULL,
    operator    TEXT NOT NULL CHECK (operator IN ('>', '<', '>=', '<=', '=')),
    threshold   DOUBLE PRECISION NOT NULL,
    severity    TEXT NOT NULL DEFAULT 'warning',
    message_tpl TEXT NOT NULL,
    cooldown_s  INT  NOT NULL DEFAULT 300,  -- seconds between re-alerts
    is_active   BOOLEAN NOT NULL DEFAULT true
);

-- Example rules
INSERT INTO alert_rules (metric, operator, threshold, severity, message_tpl) VALUES
    ('temperature', '>',  30.0, 'warning',  'Temperature exceeded 30°C on {device_id}'),
    ('temperature', '>',  35.0, 'critical', 'CRITICAL: Temperature above 35°C on {device_id}'),
    ('fuel',        '<',  50.0, 'warning',  'Generator fuel below 50 liters on {device_id}'),
    ('heartbeat',   '<',   1.0, 'critical', 'Device {device_id} has gone offline');

Alert engine Python service

python
# bridge/alert_engine.py
"""
Evaluate telemetry messages against alert rules.
Called from the bridge after each telemetry insert.
"""

import logging
from datetime import datetime, timezone, timedelta
import psycopg2
from psycopg2.extras import Json

log = logging.getLogger(__name__)


def check_and_raise_alerts(cursor, device_id: str, metric: str, value: float):
    """Check value against all matching rules and insert alerts if triggered."""

    # Fetch matching active rules
    cursor.execute(
        """
        SELECT id, operator, threshold, severity, message_tpl, cooldown_s
        FROM   alert_rules
        WHERE  is_active = true
        AND    metric    = %s
        AND    (device_id IS NULL OR device_id = %s)
        """,
        (metric, device_id),
    )
    rules = cursor.fetchall()

    for rule_id, operator, threshold, severity, message_tpl, cooldown_s in rules:
        # Evaluate condition
        triggered = eval_condition(value, operator, threshold)
        if not triggered:
            continue

        # Check cooldown — do not spam if already alerted recently
        cursor.execute(
            """
            SELECT created_at FROM alerts
            WHERE  device_id = %s
            AND    alert_type = %s
            AND    resolved_at IS NULL
            AND    created_at > NOW() - INTERVAL '%s seconds'
            ORDER  BY created_at DESC LIMIT 1
            """,
            (device_id, f"rule:{rule_id}", cooldown_s),
        )
        if cursor.fetchone():
            continue  # Still in cooldown

        # Format message
        message = message_tpl.replace("{device_id}", device_id)
        message = message.replace("{value}", str(round(value, 2)))

        # Insert alert
        cursor.execute(
            """
            INSERT INTO alerts
                (device_id, alert_type, severity, message, payload)
            VALUES (%s, %s, %s, %s, %s)
            """,
            (
                device_id,
                f"rule:{rule_id}",
                severity,
                message,
                Json({"metric": metric, "value": value, "threshold": threshold}),
            ),
        )
        log.warning(f"ALERT [{severity}]: {message}")


def eval_condition(value: float, operator: str, threshold: float) -> bool:
    ops = {'>': value > threshold, '<': value < threshold,
           '>=': value >= threshold, '<=': value <= threshold,
           '=': value == threshold}
    return ops.get(operator, False)

"""REST API over the IoT Postgres database."""

import os
from datetime import datetime
from typing import Optional

import asyncpg
from fastapi import FastAPI, Query, HTTPException
from fastapi.middleware.cors import CORSMiddleware
from dotenv import load_dotenv

load_dotenv()
app = FastAPI(title="IoT Telemetry API", version="1.0")

# Allow web dashboard on any origin (restrict in production)
app.add_middleware(
    CORSMiddleware,
    allow_origins=["*"],
    allow_methods=["GET"],
    allow_headers=["*"],
)

DB_DSN = (
    f"postgresql://{os.getenv('DB_USER','iot')}:{os.getenv('DB_PASSWORD','iot')}"
    f"@{os.getenv('DB_HOST','localhost')}:{os.getenv('DB_PORT','5432')}"
    f"/{os.getenv('DB_NAME','iot')}"
)

@app.on_event("startup")
async def startup():
    app.state.pool = await asyncpg.create_pool(DB_DSN, min_size=2, max_size=10)

@app.on_event("shutdown")
async def shutdown():
    await app.state.pool.close()


@app.get("/devices")
async def list_devices():
    """All registered devices."""
    async with app.state.pool.acquire() as conn:
        rows = await conn.fetch(
            "SELECT id, device_id, site, building, floor, room, device_type, is_active "
            "FROM devices ORDER BY site, building, floor, room"
        )
    return [dict(r) for r in rows]


@app.get("/state")
async def current_state(site: Optional[str] = None):
    """Last known reading for every device."""
    async with app.state.pool.acquire() as conn:
        if site:
            rows = await conn.fetch(
                """
                SELECT s.device_id, s.metric, s.value_num, s.unit, s.last_seen
                FROM   device_state s
                JOIN   devices d ON d.device_id = s.device_id
                WHERE  d.site = $1
                ORDER  BY s.last_seen DESC
                """, site,
            )
        else:
            rows = await conn.fetch(
                "SELECT device_id, metric, value_num, unit, last_seen "
                "FROM device_state ORDER BY last_seen DESC"
            )
    return [dict(r) for r in rows]


@app.get("/telemetry/{device_id}")
async def device_telemetry(
    device_id: str,
    metric:    Optional[str] = None,
    hours:     int = Query(default=24, ge=1, le=168),
    limit:     int = Query(default=500, ge=1, le=5000),
):
    """Recent telemetry history for a device."""
    async with app.state.pool.acquire() as conn:
        rows = await conn.fetch(
            """
            SELECT received_at, metric, value_num, unit
            FROM   telemetry
            WHERE  device_id    = $1
            AND    received_at  > NOW() - ($2 * INTERVAL '1 hour')
            AND    ($3::text IS NULL OR metric = $3)
            ORDER  BY received_at DESC
            LIMIT  $4
            """,
            device_id, hours, metric, limit,
        )
    return [dict(r) for r in rows]


@app.get("/alerts")
async def active_alerts(severity: Optional[str] = None):
    """All unresolved alerts."""
    async with app.state.pool.acquire() as conn:
        rows = await conn.fetch(
            """
            SELECT id, created_at, device_id, alert_type, severity, message
            FROM   alerts
            WHERE  resolved_at IS NULL
            AND    ($1::text IS NULL OR severity = $1)
            ORDER  BY created_at DESC
            LIMIT  200
            """, severity,
        )
    return [dict(r) for r in rows]

What you must add before this runs in a real building

The stack above is fully functional and educational. Before deploying it in a hotel, hospital, or industrial facility, you must address these areas. Skipping them is not a shortcut — it is a liability.

2. Connection pooling with PgBouncer

Every bridge reconnect attempts a new Postgres connection. At scale, direct connections exhaust Postgres capacity. Add PgBouncer in transaction-pool mode between your services and Postgres.

python
# Add a queue buffer in the bridge to decouple MQTT receive rate
# from Postgres write rate — prevents message loss under DB load spikes
import queue, threading

message_queue = queue.Queue(maxsize=10000)

def on_message(client, userdata, msg):
    """Fast path: just enqueue."""
    try:
        message_queue.put_nowait((msg.topic, msg.qos, msg.payload))
    except queue.Full:
        log.error("Message queue full — dropping message. Scale the writer.")

def db_writer_thread(db_conn):
    """Dedicated thread for DB writes, drains the queue."""
    while True:
        topic, qos, raw = message_queue.get()
        payload = parse_payload(raw)
        try:
            with db_conn.cursor() as cur:
                insert_telemetry(cur, topic, qos, payload)
                upsert_device_state(cur, topic, payload)
            db_conn.commit()
        except Exception as e:
            log.error(f"Writer error: {e}")
            db_conn.rollback()
        finally:
            message_queue.task_done()

# Start the writer thread
threading.Thread(target=db_writer_thread, args=(db_conn,), daemon=True).start()
